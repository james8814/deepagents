## 关键问题与修复

### P0级问题（已100%修复）

| # | 问题 | V4状态 | V5修复方案 | 验证 |

| 3 | **backend.read()类型不匹配** | 假设返回对象 | 改用download_files()检测存在性 | ✅ 测试通过 |

### 详细修复说明

#### P0-1: 锁内存泄漏修复

**问题**: V4使用普通dict存储线程锁，长期运行会导致OOM
```python
# V4（有泄漏）
self._locks: dict[int, threading.Lock] = {}
# 锁只增不减，runtime对象销毁后锁仍存在
```

**修复**: 使用WeakKeyDictionary
```python
# V5（已修复）
_state_locks: weakref.WeakKeyDictionary[Any, threading.Lock] = weakref.WeakKeyDictionary()
# runtime对象被垃圾回收时，锁自动释放
```

**验证**:
- 单元测试验证锁的创建和释放
- 并发测试验证线程安全性
- 内存分析确认无泄漏

#### P0-2: backend.read()返回类型修复

**问题**: V4假设read()返回带.found属性的对象
```python
# V4（错误假设）
read_result = backend.read(path)
if read_result.found:  # ❌ 实际返回str
    previous_size = sum(len(line) for line in read_result.contents)
```

**修复**: 使用download_files()检测
```python
# V5（正确实现）
responses = backend.download_files([path])
is_overwrite = responses[0].error is None
previous_size = len(responses[0].content) if is_overwrite else None
```

**验证**:
- 与FilesystemBackend集成测试通过
- 与StateBackend集成测试通过
- 与CompositeBackend集成测试通过

## 设计决策与辩证分析

### 关键设计争议解决

#### 1. 类 vs 函数

**争议**: V4使用7个类（企业级Java风格），简化方案使用3-4个函数（Pythonic风格）

**辩证分析**:
- 类设计的优势：扩展性强、符合SOLID原则
- 函数设计的优势：简洁、易理解、与DeepAgents基座一致

**决策**: 采用函数设计

**理由**:
1. DeepAgents现有代码大量使用函数模式
2. 当前需求简单，无需复杂扩展机制
3. 函数设计更易于测试和维护
4. 220行 vs 400行，减少45%代码

#### 2. 规则注册 vs if-elif

**争议**: V4使用StrategyRule注册机制，简化方案使用if-elif链

**辩证分析**:
- 规则注册的优势：支持第三方扩展、符合OCP
- if-elif的优势：简单直观、易于调试

**决策**: 采用if-elif链

**理由**:
1. 当前backend类型有限（4种），扩展需求不明确
2. if-elif链更易于理解和维护
3. 未来如需扩展，可轻松重构为规则注册

#### 3. 能力检测方式

**争议**: hasattr vs isinstance vs 行为检测

**决策**: 使用isinstance + 类型映射

**理由**:
1. 类型安全，IDE支持好
2. 无副作用（不创建测试文件）
3. 与DeepAgents基座代码风格一致
